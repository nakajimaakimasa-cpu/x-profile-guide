# Goal Engineering by Addness

どうも！アドネス株式会社のみかみです。

Context Engineering、みんな試行錯誤してる。CLAUDE.md、Skills、MCP、RAG、Memory。日進月歩で手法が更新されて、各自がおのおので最適解を模索してる。でも「これが決定版」がない。収束しない。

$ この記事が、その答えを出す。

結論から言う。Context Engineeringが収束しないのは、上流が設計されていないから。「AIに何を渡すか」は「何を達成したいか」が決まらないと決まらない。ゴールの構造的な設計——Goal Engineering——が先にあれば、コンテキストはその関数として自動的に決まる。

$ Addnessは、このGoal Engineeringをツリー構造で実装している。以下、5つの設計判断でその全体像を解説する。

---

## なぜContext Engineeringは収束しないのか

Prompt Engineering（2023年）は「どう聞くか」の最適化。ほぼ解かれた。プロンプトの書き方で差がつく時代は終わった。

Context Engineering（2025年）は「何を渡すか」の最適化。こっちは未解決。CLAUDE.mdを書く人、MCPでツールを繋ぐ人、RAGでドキュメントを食わせる人、Skillsでワークフローを定義する人。全員がそれぞれのアプローチで試行錯誤している。手法は週単位で増えるのに、収束点が見えない。

$ 収束しない理由は構造的なもの。Context Engineeringには評価関数がない。

CLAUDE.mdを100行書いた。それが良いCLAUDE.mdかどうか、どう判定する？ MCPで10個のツールを繋いだ。その10個が正しい10個だとなぜ言える？ RAGの検索精度を上げた。そもそも検索すべき情報がそれで合っているとなぜ分かる？

「良い」「正しい」「合っている」。これらは全部、ゴールに対する相対評価。ゴールが形式化されていなければ、コンテキストの良し悪しを判定する基準自体が存在しない。

$ 評価関数のない最適化は収束しない。当たり前の話。

ゴールがツリーで構造化されていれば変わる。ツリーの各ノードが「このタスクの達成に必要な情報」を定義する。コンテキストがゴール構造から演繹的に導出される。「正しいコンテキスト」が初めて定義可能になる。

$ Goal EngineeringはContext Engineeringに評価関数を与える。上流を構造化すれば、下流は収束する。

Addnessはこの難問に5つの軸でアプローチしている。

---

## 1. ゴールをツリーで分解する

$ 「売上を上げたい」から実行可能なタスクリストを作れる人は、ほぼいない。

これは能力の問題じゃなく、構造の問題。「売上を上げたい」は抽象度が高すぎて、そこから「今日やるべきこと」までの距離が遠すぎる。人間の脳は3階層以上の分解を同時に処理できない。だからみんなゴールを立てた後、いきなりタスクリストを書く。ゴールとタスクの間に論理的な接続がないまま。

OKRはこの問題を「Objective → Key Result」の2階層で部分的に解いた。でもKey Resultからタスクへの分解は完全に人間任せ。「売上30%増」がObjectiveで「新規リード数を月200件にする」がKey Result。で、そのために具体的に何をやるの？　ここが空白のまま。

$ Addnessは、ゴールからタスクまでの全階層をAIがツリーとして生成する。

話すだけでいい。「3ヶ月で月商を1.5倍にしたい。今のボトルネックは新規獲得チャネルの不足」。これだけ話せば、ゴール → サブゴール → タスク → サブタスクのツリーが構造的に生成される。各ノードには達成条件がつく。曖昧な分解を許さない。

$ OKRは「ゴール→指標」の2階層で止まる。Addnessは「ゴール→末端タスク」まで全階層を一気通貫で繋ぐ。ここが根本的に違う。

---

## 2. ゴールに繋がってないタスクを存在させない

$ 既存のタスク管理ツールは「タスクの管理」はできるが、「タスクの正しさ」は検証しない。

Notionにタスクを100個並べられる。Asanaで進捗を可視化できる。Jiraでスプリントを回せる。でもその100個のタスクのうち、ゴールに本当に繋がっているのが何個かは、どのツールも教えてくれない。

Addnessは逆の設計をしている。

$ ツリーに載っていないタスクは、表示しない。

全てのタスクはゴールからのツリー上に存在する。ツリーの外にタスクを作ることができない。これは制約であり、最大の機能。「面白そうだから」「なんとなく」で追加されたゴミタスクが構造的に排除される。

Gitで言えば、mainブランチに繋がっていないブランチは死んだブランチ。どれだけコミットを積んでもマージされない。Addnessのタスクも同じ。ゴールに繋がっていないタスクは存在しない。

$ Notion/Asana/Jiraは「タスクを効率よく消化する」ツール。Addnessは「正しいタスクだけが存在する」ツール。解いている問題が違う。

## 3. 「今日やること」をAIが決める

ゴールが分解されてツリーができても、「で、今日はどこからやるの？」が残る。

$ ツリーの末端タスクが50個あったら、人間は50個の中から選ぶだけで疲弊する。

GTDの「次にやることリスト」もOmniFocusの「Today」表示も、結局は人間が自分で優先度を判断している。判断コスト自体が生産性を食っている。

$ Addnessは、ツリーの構造からAIが「今やるべきもの」を自動で選ぶ。

ロジックはツリーの性質そのもの。末端タスクのうち、親ノードの達成に最もインパクトが大きく、依存関係がクリアなものが出てくる。人間が優先順位を考える必要がない。

結果、朝起きて「今日何しよう」と考える時間がゼロになる。ツリーが正しければ、出てきたものをやるだけでゴールに近づく。

$ タスク管理において「何をやるか」を考える時間は、本質的に無駄。Addnessはこの意思決定コストをゼロにする設計。

---

## 4. ツリーは生き物

$ 一度作ったツリーが正しいままであることは、まずない。

市場が変わる。前提が崩れる。やってみたら想定と違った。これが現実。OKRが四半期ごとの見直しをルールにしているのは、計画が古くなることを前提にしているから。

でも四半期は遅すぎる。AI時代は週単位で前提が変わる。

$ Addnessは、ツリーの正しさを常にAIが検証し続ける。

タスクの実行結果がツリーにフィードバックされる。達成条件を満たしたノードは完了する。前提が変わったサブゴールはAIが検知して修正を提案する。ツリーは「一度作ったら終わり」の静的な計画書ではなく、常に更新される生きた構造体。

$ OKRの「四半期見直し」、プロジェクト管理の「計画変更会議」。全部、構造が自動で更新されないことの代償。Addnessはこの代償をゼロにしている。

---

## 5. 個人のツリーが組織のツリーに接続する

$ 1人のゴールツリーは簡単に作れる。300人のゴールツリーを統合するのは全く別の問題。

社長のゴール「年商10億」の下に、営業部のゴール「月間受注100件」があり、その下に営業メンバーのゴール「週10件の商談」があり、その下にタスク「リスト作成」「アポ獲得」がある。この構造が、全員分、一本のツリーで繋がっている必要がある。

$ 既存のツールでこれを実現できるものはない。

Asanaのプロジェクトはフラット。Notionのデータベースは個人の島。OKRツールはゴール→指標までで、その下のタスク層は別ツールに丸投げ。結果、社長のゴールと現場のタスクが切断される。「今やっていることが会社のゴールに繋がっているか」を現場が分からない。

$ Addnessは、個人のツリーを組織のツリーのサブツリーとして接続する。

社長のゴールツリーの末端が、部門長のゴールツリーの頂点になる。部門長のツリーの末端が、メンバーのツリーの頂点になる。再帰的な階層構造。全員が自分のツリーを見れば、それが組織のゴールのどこに繋がっているか分かる。

$ これは348人の組織で実運用している構造。社長が全員に直接指示を出す組織は成長しない。ゴールが明確で、ツリーで分解されて、末端から自律的に動く組織だけが成長する。200人以上の組織を経営してきて、これは確信している。

## まとめ: Addnessが解いている5つの問題

1. ゴールの構造的分解 → 全階層を一気通貫でツリー生成
2. タスクの正しさの担保 → ツリー外のタスクを存在させない設計
3. 意思決定コストの排除 → 今やるべきことをAIが自動選出
4. 計画の陳腐化の防止 → ツリーをAIが常に検証・更新
5. 組織スケール → 個人のツリーを組織のツリーに再帰的に接続

$ 既存のツールは、この5つのうちどれも解いていない。

Notionは書ける。Asanaは管理できる。OKRはゴールを立てられる。でも「ゴールから末端タスクまでの構造的な接続」を保証し、「タスクの正しさ」を検証し、「今やるべきこと」を自動で出し、「計画を動的に更新」し、「組織全体に展開」するツールは、Addness以外にない。

$ 特許取得済み。

## 「アドる」の正体

> 「ググるとわかる。アドればできる。」

疑問がある。ググる。分かる。
ゴールがある。アドる。できる。

$ 「ググる」はGoogleが人間に与えた動詞。「アドる」はAddnessが人間に与える動詞。

ググるの中身を知らなくていい。PageRankもクローリングもインデックスも、ユーザーは意識しない。検索窓に打てば答えが出る。

アドるも同じ。ゴールのツリー分解も、タスクの正当性検証も、優先度の自動選出も、ツリーの動的更新も、組織への接続も、ユーザーは意識しない。ゴールを話せば、できる。

$ この記事で語った5つの設計判断は、全部「アドる」の裏側で動く仕組みにすぎない。

- Googleは「知る」のボトルネックを解消した
- OpenAIは「生成する」のボトルネックを解消した
- Addnessは「できる」のボトルネックを解消する

全タスクがゴールに繋がっている。「今この瞬間が、目的にプラスに寄与している」と感じられる状態。これを全人類が常にキープできるような世界にしたい。

$ Goal Engineering by Addness。これで世界と戦う。
